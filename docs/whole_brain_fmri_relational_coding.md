# Whole-Brain fMRI Relational Coding

This document explains how the repository computes *relational coding* distances between task and rest fMRI responses for every region of interest (ROI). The goal is to quantify how similarly a subject (or group average) patterns of voxel activity respond to narrative movie clips versus nearby resting-state segments.

---

> **Architecture note.** This write-up predates the "flatten-flows" refactor. The
> algorithm is unchanged, but the code now lives in plain functions and modules rather
> than the old `relational_coding/` class hierarchy dispatched through `FlowManager`:
> shared algorithms in `rc_core.py`, data loaders in `data_access.py`, and one module
> per analysis under `flows/` (each exposing a top-level `run(...)`). Paths come from
> `settings.py`. Mentions of `FlowManager._relational_coding`,
> `FmriRelationalCoding`, `RelationalCodingBase`, or `config` below map onto these.

## Inputs and Dimensions

| Artifact | Shape / Structure | Source |
| --- | --- | --- |
| ROI-level tabular data per subject | DataFrame with columns `feat_0 … feat_{n}` (voxel indices tied to the ROI), plus metadata columns `timepoint`, `y` (clip id 1–14), and optional `Subject`. Each row encodes a single TR sample. | Produced by `data_normalizer.Voxel2Roi` using the Schaefer2018 300-ROI / 7-network atlas. |
| Averaged ROI data (optional) | Same columns as above, but averaged across a predefined subject group (e.g., `_GROUP1`). | Generated previously and stored under `config.SUBNET_DATA_AVG`. |
| Static metadata | JSON-driven lists of ROI names, subject IDs, clip ordering, and clip name ↔ index mappings. | `data_center/static_data/static_data.json`, loaded at runtime via `StaticData.inhabit_class_members()`. |
| Configuration flags | `avg_data`, `group`, `shuffle`, and optional filtering/decomposition/movies distance flags forwarded from CLI helpers. | Provided when calling `FlowManager.execute`. |

Typical dimensionalities:

- Number of ROIs: 300 (Schaefer2018 atlas; see `Voxel2Roi.load_voxel_mapping_file` for the `roi=300` setting).
- Number of task clips: 14 (indices `1..14`, named via `StaticData.CLIP_MAPPING`).
- Resting-state TR samples evaluated per clip alignment: 19 (generated by `RelationalCodingBase.rest_between_tr_generator`).
- Feature count per ROI: varies with ROI size; each `feat_k` column maps to a voxel belonging to the ROI as defined in the atlas.

## Step-by-Step Algorithm

1. **Initialization**
   - `FlowManager.execute` is called with `flow_type=FlowType.RELATIONAL_CODING`, the ROI name, a `DataType` (only `FMRI` is supported here), and flags such as `avg_data`, `group`, and `shuffle`.
   - `FlowManager` instantiates `FmriRelationalCoding` and forwards the parameters.

2. **Data Loading**
   - When `avg_data=False`, `subject_flow` iterates over every subject id enumerated by `StaticData.SUBJECTS`.
   - For each subject:
     - Task data are loaded from `Schaefer2018_SUBNET_TASK_DF/<subject>/<roi>.pkl`.
     - Rest data are loaded from `Schaefer2018_SUBNET_REST_DF/<subject>/<roi>.pkl`.
   - When `avg_data=True`, `avg_data_flow` loads group-wise averages from `Schaefer2018_SUBNET_AVG_{mode}{group}` once per ROI.
   - Both loading methods call `RelationalCodingBase.load_roi_data` / `load_avg_data`, which also validate the ROI name against `StaticData.ROI_NAMES`.

3. **Clip Vector Construction**
   - For every rest timepoint `tr ∈ [0, 18]`, `relation_distance` builds a dictionary of 28 vectors (14 task + 14 rest) via `get_clip_vectors`.
   - `get_clip_vectors`:
     - Loops over clip indices `1..14`.
     - Pulls the most recent task TR for the clip (`get_single_tr_vector` with `timepoint=-1`).
     - Pulls the rest vector aligned to the current rest TR.
     - Keys the results as `<clip_name>_task` and `<clip_name>_rest`.
   - Each vector is an ordered list of ROI voxel intensities (`feat_x` columns) for that TR.

4. **Correlation Stack Construction**
   - `correlate_current_timepoint` receives the dictionary of vectors and turns it into a Pandas DataFrame.
   - Columns are reordered using `StaticData.CLIPS_ORDER` and `StaticData.REST_ORDER` so that all task clips precede rest clips.
   - Optional preprocessing (configured via kwargs but typically disabled in this flow):
     - `shuffle_rest`: randomize the ordering of rest columns for permutation control.
     - `filtering`: apply low-pass filtering (order/cut-off configurable).
     - `decomposition`: reduce dimensions with PCA (`n_components=0.1` fraction).
     - `movie_distances`: project clip responses into a movie-distance vector space via `distances_utils`.
   - A full correlation matrix (`df_corr`) is computed across all 28 vectors.

5. **Relational Distance Computation**
   - `df_corr` is split into task-vs-task (`clip_cor`) and rest-vs-rest (`rest_cor`) quadrants.
   - `MatrixOperations.drop_symmetric_side_of_a_matrix` flattens each quadrant into a vector containing the upper-triangular (non-redundant) correlation coefficients.
   - These flattened vectors are combined into a 2-column DataFrame (`general_corr`) labelled `task` and `rest`.
   - The Pearson correlation between the two columns is computed; the resulting scalar (rounded to 0.001) is the *relational coding distance* for that rest TR.
   - The function also returns the full correlation matrix (with NaNs replaced by 1) so downstream consumers can inspect the full structure if needed.

6. **Result Aggregation and Persistence**
   - For per-subject runs, `subject_flow` accumulates a list of 19 distances (one per rest TR) and stores it under the subject’s key in a dictionary.
   - For averaged runs, `avg_data_flow` stores the 19-value list under the `'avg'` key and optionally adds `'avg correlation'` matrices for inspection.
   - Results are serialized as pickles using `data_normalizer.utils.dict_to_pkl`.
     - Subject-level outputs reside in `RELATIONAL_CODING/relational_coding/<roi>.pkl`.
     - Average-level outputs are written to `RELATIONAL_CODING/relational_coding_avg{group}/<roi>.pkl`.
     - Shuffle-control runs save to `RELATIONAL_CODING/relational_coding_shuffle_rest/<roi>.pkl`.
   - Existing result files short-circuit future runs to avoid recomputation.

## Operational Notes and Gotchas

- **ROI Validation**: Passing an ROI name not listed in `StaticData.ROI_NAMES` raises an exception before any disk I/O happens.
- **Data Availability**: The flow assumes prior preprocessing steps (`VoxelExtraction`, `Voxel2Roi`, averaging) have already produced the necessary `.pkl` tables on the external drive defined in `config`.
- **Determinism**: Unless `shuffle_rest=True`, the algorithm is deterministic. Any stochastic controls (rest shuffling or downstream random seeds) rely on NumPy’s global RNG state.
- **Performance Considerations**:
  - Each ROI × subject combination loads two pickles and computes 19 correlation matrices; batching across ROIs can be I/O heavy.
  - Optional filtering or PCA increases compute time and memory footprint but shares the same API.
- **Visualization Hooks**: After generating results, `visualizations/plot_relational_coding.py` can render per-ROI summaries; see the helper functions in `main.py`.

## How to Reproduce

1. Ensure static metadata are loaded once at startup:
   ```python
   from data_center.static_data.static_data import StaticData
   StaticData.inhabit_class_members()
   ```
2. Run the helper in `main.py`, call the flow directly, or use the CLI. Examples:
   ```python
   import flows.fmri_relational_coding as fmri_relational_coding

   fmri_relational_coding.run(
       'RH_Default_pCunPCC_6',
       avg_data=False,   # per-subject (True -> group average)
       group='',         # group suffix (used only when avg_data=True)
       shuffle=False,    # permutation control
   )
   ```
   ```bash
   python -m cli relational-coding --roi RH_Default_pCunPCC_6
   ```
3. Inspect outputs under the directories declared in `settings.FMRI_RELATION_CODING_RESULTS*`.
